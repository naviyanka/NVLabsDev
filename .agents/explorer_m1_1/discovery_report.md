# NEXUS Frontend Discovery Report

This report summarizes the findings of the React frontend investigation for the NEXUS application located under `src/Nexus.Frontend`. 

---

## 1. Unused and Dead Files

A large portion of files in the frontend codebase are completely unused. They have no incoming imports in the actual active execution tree of the application.

### Logically Unused Source Files
The following files under `src/` are dead and can be safely deleted:
1. `src/routes/settings_old.tsx`: An obsolete version of the settings route. It is imported only by the autogenerated `routeTree.gen.ts` because it sits in the `routes/` directory, but it is never referenced or linked to in the application.
2. `src/components/dashboard/WidgetRegistry.tsx`: A completely unused widget registry file containing dummy render methods (`() => null`). It is not imported by any other file.
3. `src/lib/api/example.functions.ts`: A sample server-side function (`createServerFn`) that is never imported or used.
4. `src/lib/config.server.ts`: Configuration file imported *only* by `example.functions.ts`. Because the latter is dead, `config.server.ts` is transitively dead.
5. `src/hooks/use-mobile.tsx`: A custom hook imported *only* by `src/components/ui/sidebar.tsx` (which is itself unused).

## 2. Dead and Unused Code (Within Files)

Even within active files, there are obsolete helper functions, mock API interfaces, and imports that are never called:

* **Unused PowerShell HTTP client methods (`src/api/client.ts`):**
  * `createPsSession` (lines 417-426)
  * `destroyPsSession` (lines 428-434)
  * `runPowerShellClient` (lines 436-487)
  * *Reason:* The application's PowerShell pages (`src/routes/powershell.tsx` and `src/themes/horizon/pages/HorizonPowerShell.tsx`) interact with the backend terminal using direct WebSockets (`ws://<host>/api/terminal/ws`) instead of HTTP requests.
* **Unused Mock API Helper Functions (`src/api/mock.ts`):**
  * Functions like `getServers`, `addServer`, `getServerDetail`, `killProcess`, `controlService`, `getDisks`, `getVolumes`, `getPerformanceHistory`, `getUpdates`, `installUpdate`, `getScheduledTasks`, `runTask`, `getInstalledApps`, `uninstallApp`, `getRolesAndFeatures`, `getPlugins`, `togglePlugin`, and `runPowerShell` are completely uncalled.
  * *Reason:* Active route components fetch data by invoking the actual client methods from `src/api/client.ts` directly. They only import types and static data constants (`MOCK_SERVERS`, `MOCK_APPS`) from `mock.ts`, leaving all mock implementation functions dead.

## 4. Code Duplication and Layout Analysis

### The "Horizon" Theme Architecture (Messy Logic / Duplication)
NEXUS supports multiple themes. Most themes (`cyberpunk`, `dark`, `infrared`, `light`, `slate`, `stealth`) are implemented cleanly via CSS selectors linked dynamically in `__root.tsx`.

However, the **Horizon** theme is implemented as a parallel codebase mirror under `src/themes/horizon/pages/`. This leads to major duplication:
* **Mirror pages:** For each page (dashboard, login, plugins, powershell, servers, settings, and plugin details), there are two completely separate files (e.g., `src/routes/servers.tsx` vs `src/themes/horizon/pages/HorizonServers.tsx`).
* **Logic Duplication:** Each pair of pages duplicates React state definitions, client API invocations (`useEffect`, `useState`), alert notifications (`toast`), and CRUD handlers.
* **Conditional rendering:** Every route file contains an inline conditional check at the component root:
  ```typescript
  if (theme === 'horizon') return <HorizonPage />;
  ```

### Code Layout
* The layout uses TanStack Start with Vinxi under the hood. 
* Configured using `@lovable.dev/vite-tanstack-config` with custom server settings (wildcard SSL PFX configuration, reverse proxying `/api` and `/hub` endpoints to `https://localhost:5011`).
* Unused UI files clutter `src/components/ui/` and should be pruned to keep the bundle and development context light.

## 5. Build and Run Configurations

### Build Command
The frontend builds successfully with the following command executed in the `src/Nexus.Frontend` directory:
```bash
npm run build
```
*(Alternatively, `bun run build` can be used since `bun.lock` is present).*

**Verification Output Snippet:**
```

transforming...✓ 151 modules transformed.
rendering chunks...
computing gzip size...
dist/server/server.js                                       3.27 kB │ gzip:  1.42 kB
...
✓ built in 2.03s
```
The output compiles client assets under `dist/client/` and the SSR entry point under `dist/server/server.js`.

### Run Commands
1. **Development Mode (with Hot Module Replacement):**
   ```bash
   npm run dev
   ```
   Starts a dev server at `https://0.0.0.0:443` proxying to `https://localhost:5011`.
2. **Production Mode (Running the Compiled Server):**
   The compiled server is executed via:
   ```bash
   node dist/server/server.js
   ```
   *(Or via `npm run preview` to locally test the production build).*
